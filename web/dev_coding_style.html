Coding Style and Documentation

The project follows particular coding conventions in order to create an uniform coding style throughout all classes. In order to describe the major facets regarding the coding style, we will reference the following elements individually: naming, spacing/formatting, and comments.

Naming:

  • Variable and function names should be descriptive.

  • Example: int width, ColorData pixel

  • Function names should be camel-case (i.e., the first word is in lowercase and all subsequent words are in uppercase)

  • Example: myFunctionName( )

  • Local variable names should be in all lower case, and may use an underscore to separate words.

  • Example: myvariablename or my_variable_name

  • Class names should capitalize the first letter of every word

  • Example: MyClassName.cpp

  • Class data members should have the prefix ‘m_’

  • Example: int m_width

  • Constant variable names should be in all uppercase

  • Example: int CONSTANTVAR

Spacing/Formatting:

The indentation size used was equivalent to 8 spaces. It would be beneficial if contributors to the project adjusted their text editor so that the <tab> key creates 8 spaces. This indentation scheme should be utilized in the following situations:

  • Starting a function definition

myFunction() {

                                        int x;

  • Following a for/while/do-while statements

for(int i = 0; i < 10; i--) {

                                int x;

  • Following if/else conditional statements

if(true) {

                                int x;

  • Following private/public/protected definitions in the header file

Class className {

Public:

                                int x;

Classes:

  1. There should be no indentation for the public, private, and protected specifiers.
  2. The visibility specifiers should be preceded by a white space, except for the first one listed.
  3. No vertical white space should follow the visibility specifiers
  4. The name of the parent class should be on the same line as the derived class

Functions:

  1. There should be a white space before the open curly brace
  2. The closing curly brace should be on its own line at the end
  3. There should be no spaces between parentheses and parameters
  4. There should be one space after each comma in the parameter list

Conditionals:

  1. Rules 1 and 2 from functions also apply to conditionals
  2. There may be zero or one white space between the conditional statement and the parentheses. For example:

                                                if( true ) and if (true) are both valid

  1. An else statement should be on the same line as the closing curly brace of the previous if statement, followed by its open curly brace. For example:

If (true) {

} else {

  1. Note that the open curly bracket is always included on the same line as the statement. Additionally, the closing curly bracket should be the last line on its own.
  2. There should be a space between a closing parenthesis and the opening curly bracket
  3. In conditionals, loops, and function, there should be no spaces between parentheses and the parameters

Comments

        

Commenting is an essential component to making code readable and easy to understand. Comments should be used to supplement descriptive naming conventions and not as a substitution. Refer to the following guidelines to determine when comments are appropriate:

  1. Include comments above functions when the name is not able to fully convey the actions that it performs
  2. Include line comments to describe particularly complex code segments, or code whose function is not readily apparent
  3. Do not include comments that describe the purpose of a variable
  4. Every .h file should have a brief comment describing the class
  5. TODO comments should be used to mark sections that need to be revised and should include a brief description
